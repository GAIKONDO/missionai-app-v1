# コントリビューションガイド

> **📋 ステータス**: アクティブ（コントリビューションガイド）  
> **📅 最終更新**: 2025-12-11  
> **👤 用途**: コントリビューション方法、コーディング規約、プルリクエストの手順

## 概要

MissionAIプロジェクトへのコントリビューションを歓迎します。このガイドでは、コントリビューションの方法とプロジェクトの標準を説明します。

## コントリビューションの種類

以下のようなコントリビューションを歓迎します：

- 🐛 **バグ修正**: バグの報告と修正
- ✨ **新機能**: 新機能の提案と実装
- 📝 **ドキュメント**: ドキュメントの改善
- 🎨 **UI/UX**: ユーザーインターフェースの改善
- ⚡ **パフォーマンス**: パフォーマンスの最適化
- 🔧 **リファクタリング**: コードの改善

## 開発環境のセットアップ

コントリビューションを始める前に、開発環境をセットアップしてください。

詳細は[開発環境セットアップガイド](../setup/SETUP_GUIDE.md)を参照してください。

### クイックスタート

```bash
# 1. リポジトリのクローン
git clone <repository-url>
cd app40_MissionAI

# 2. 依存関係のインストール
npm install

# 3. 環境変数の設定
cp local.env.example local.env  # 存在する場合
# local.envファイルを編集

# 4. 開発サーバーの起動
npm run tauri:dev
```

## コーディング規約

### TypeScript/JavaScript

#### スタイル

- **インデント**: 2スペース
- **セミコロン**: 使用する
- **クォート**: シングルクォート（`'`）を使用
- **行の長さ**: 100文字以内（可能な限り）

#### 命名規則

- **変数・関数**: `camelCase`
- **クラス・コンポーネント**: `PascalCase`
- **定数**: `UPPER_SNAKE_CASE`
- **ファイル名**: `kebab-case`（コンポーネントは`PascalCase.tsx`）

#### 例

```typescript
// ✅ 良い例
const userName = 'John';
function getUserData() { }
class UserProfile { }
const MAX_RETRIES = 3;

// ❌ 悪い例
const user_name = 'John';
function get_user_data() { }
class user_profile { }
```

### Rust

#### スタイル

- **インデント**: 4スペース（タブ）
- **命名規則**: Rustの標準規約に従う
  - **変数・関数**: `snake_case`
  - **型・構造体**: `PascalCase`
  - **定数**: `UPPER_SNAKE_CASE`

#### 例

```rust
// ✅ 良い例
let user_name = "John";
fn get_user_data() { }
struct UserProfile { }
const MAX_RETRIES: u32 = 3;

// ❌ 悪い例
let userName = "John";
fn getUserData() { }
struct user_profile { }
```

### コメント

- **日本語**: コメントは日本語で記述
- **説明**: なぜそのコードが存在するかを説明
- **TODO**: `TODO: 説明`の形式で記述

```typescript
// ✅ 良い例
// ユーザー名を取得（データベースから読み込み）
const userName = await getUserName();

// TODO: エラーハンドリングを追加
```

## ブランチ戦略

### ブランチ命名規則

- **機能追加**: `feature/機能名`
- **バグ修正**: `fix/バグ名`
- **ドキュメント**: `docs/内容`
- **リファクタリング**: `refactor/内容`

### 例

```bash
git checkout -b feature/add-user-authentication
git checkout -b fix/database-connection-error
git checkout -b docs/update-api-documentation
```

## コミットメッセージ

### 形式

```
<種類>: <簡潔な説明>

<詳細な説明（オプション）>
```

### 種類

- `feat`: 新機能
- `fix`: バグ修正
- `docs`: ドキュメント
- `style`: コードスタイルの変更
- `refactor`: リファクタリング
- `perf`: パフォーマンス改善
- `test`: テストの追加・修正
- `chore`: ビルドプロセスやツールの変更

### 例

```
feat: ユーザー認証機能を追加

- ログイン機能の実装
- セッション管理の追加
- エラーハンドリングの実装
```

## プルリクエストの手順

### 1. ブランチの作成

```bash
git checkout -b feature/your-feature-name
```

### 2. 変更の実装

- コードを実装
- テストを追加（可能な場合）
- ドキュメントを更新（必要に応じて）

### 3. コミット

```bash
git add .
git commit -m "feat: 機能の説明"
```

### 4. プッシュ

```bash
git push origin feature/your-feature-name
```

### 5. プルリクエストの作成

1. GitHubでプルリクエストを作成
2. タイトルと説明を記入
3. 関連するIssueをリンク（存在する場合）
4. レビュアーを指定

### プルリクエストのテンプレート

```markdown
## 変更内容
- 変更点1
- 変更点2

## 関連Issue
Closes #123

## テスト
- [ ] ローカルでテストを実行
- [ ] 既存のテストが通過
- [ ] 新しいテストを追加（必要に応じて）

## スクリーンショット（UI変更の場合）
<!-- スクリーンショットを添付 -->
```

## コードレビュー

### レビュアー向け

- **建設的なフィードバック**: 改善点を具体的に指摘
- **承認基準**: 
  - コードが動作する
  - テストが通過する
  - ドキュメントが更新されている
  - コーディング規約に従っている

### コントリビューター向け

- **フィードバックへの対応**: レビュアーのフィードバックに真摯に対応
- **質問**: 不明な点は質問する
- **修正**: 指摘された点を修正

---

## テスト

### テストの追加

新しい機能を追加する際は、可能な限りテストを追加してください。

**現状**: 自動テストは限定的です。手動テストを推奨します。

### テスト手順

1. **機能テスト**: 追加した機能が正常に動作するか確認
2. **回帰テスト**: 既存機能が影響を受けていないか確認
3. **エラーテスト**: エラーケースが適切に処理されるか確認

## ドキュメント

### ドキュメントの更新

コードを変更する際は、関連するドキュメントも更新してください。

**更新が必要なドキュメント**:
- API仕様書（APIエンドポイントの追加・変更時）
- 設定ドキュメント（設定の追加・変更時）
- セットアップガイド（セットアップ手順の変更時）

### ドキュメントの場所

- **設計ドキュメント**: `docs/`ディレクトリ
- **コードコメント**: コード内に記述
- **README**: プロジェクトルートの`README.md`

## 質問とサポート

### 質問方法

1. **Issue**: 機能提案やバグ報告はIssueで
2. **ディスカッション**: 一般的な質問はディスカッションで
3. **プルリクエスト**: コードレビュー中の質問はPRコメントで

### サポート

問題が発生した場合、[トラブルシューティングガイド](../troubleshooting/TROUBLESHOOTING.md)を参照してください。
